## 2. THE TRI-PILLAR ARCHITECTURE: DEEP DIVE

### 2.1 CLIDE (The Will & Memory)
*   **Path:** `core/clide/`
*   **Role:** The **Executive Branch**.
*   **Detailed Workflow:**
    1.  **Goal Parsing:** CLIDE uses a Semantic Parser to extract the "Intent" from the operator's natural language.
    2.  **Ontology Mapping:** It consults `core/clide/ontology.py` to find the most relevant "Primitive Actions".
    3.  **DAG Generation:** The `compiler.py` creates a Directed Acyclic Graph (DAG) where nodes are atomic shell commands and edges represent causal dependencies.
    4.  **Scheduler Dispatch:** The `DAGScheduler` pushes the tasks to the Redis broker, assigning them to the most suitable node based on the `operator_manifest.json`.
*   **Persistence:** Every state transition is recorded in `cap_events.db`. The system utilizes **Lamport Logical Clocks** to ensure event ordering across the distributed mesh.

### 2.2 APC-RUNTIME (The Actuator)
*   **Path:** `core/apc/`
*   **Role:** The **Execution Branch**.
*   **Detailed Workflow:**
    1.  **Pre-Execution Hash:** Before running any command, `hasher.py` generates a recursive SHA-256 hash of the target directory.
    2.  **Sandbox Isolation:** The command is executed within a restricted shell environment (via `sandbox.py`) to prevent side effects outside the project scope.
    3.  **Resource Monitoring:** APC tracks CPU, RAM, and Disk I/O during execution.
    4.  **Post-Execution Hash:** A second recursive hash is generated.
*   **Verification:** If the delta between the pre and post hashes does not match the predicted outcome of the command, the APC layer flags a **Causal Violation**.

### 2.3 PIE (The Perception & Inference)
*   **Path:** `core/pie/`
*   **Role:** The **Analytical Branch**.
*   **Detailed Workflow:**
    1.  **Causal Reconstruction:** PIE reads the `cap_events.db` and reconstructs the "Action-Event Chain" for a specific `trace_id`.
    2.  **Flavour Selection:** Depending on the query, PIE uses different inference engines:
        *   `flavours/causal.py`: "Why did this happen?"
        *   `flavours/predictive.py`: "What will happen if we run X?"
        *   `flavours/diagnostic.py`: "How do we fix Y?"
    3.  **Autopoietic Feedback:** PIE generates refactor proposals that are sent back to CLIDE for autonomous implementation.

## 4. EXAMPLE SCENARIO: DISTRIBUTED BUILD FAULT

**Scenario:** A Windows execution node fails to build a Python extension due to a missing C++ compiler.

1.  **Detection:** APC-RUNTIME on the Windows node reports an `exit_code: 1` and a `STATE_STAGNATION_ERROR` (post-hash matches pre-hash).
2.  **Notification:** The Singularity Dashboard vibrates with an `ANOMALY_DETECTED` event (Vibration: Magenta).
3.  **Introspection:** The operator (or CLIDE) calls `PIE.analyze(trace_id)`.
4.  **Inference:** PIE's `diagnostic.py` flavour identifies the error string: `error: Microsoft Visual C++ 14.0 or greater is required`.
5.  **Remediation:** PIE suggests an update to the `operator_manifest.json` to flag this node as `INCOMPLETE_COMPILER_STACK`. CLIDE then re-dispatches the build task to a Linux node with `HEAVY_COMPUTE` capabilities.
