# 🛡️ SYSTEM ONTOLOGY: `.capignore`

## 1. EXECUTIVE SUMMARY: THE CONTEXT PRUNING PROTOCOL
The `.capignore` file is a critical component of the **APC (Automated Personalized Context) Runtime**. It defines the "Semantic Boundaries" of the system by specifying which files and directories should be excluded from the system's recursive state hashing and cognitive context. In a high-frequency agentic loop, every byte processed adds to the system's "Cognitive Friction". `.capignore` serves to eliminate this friction by pruning high-entropy, transient, or irrelevant data.

Without `.capignore`, the system would attempt to hash massive `node_modules` folders or transient log files, leading to performance degradation and, more importantly, **Non-Deterministic Anomalies** in the causal ledger.

## 2. THE MECHANICS OF STATE HASHING: DEEP DIVE

### 2.1 The Hashing Algorithm (`apc/hasher.py`)
The system uses a recursive Merkle-Tree-like hashing strategy to verify filesystem integrity.
1.  **Traversal:** The hasher walks the directory tree using `os.walk` or a high-performance Rust-based equivalent.
2.  **Exclusion Matching:** Every file path is matched against the glob patterns in `.capignore`. Pattern matching is performed using the `pathmatch` library to ensure consistency with `.gitignore` standards.
3.  **Leaf Hashing:** For every non-ignored file, a SHA-256 hash of its content is calculated.
4.  **Digest Accumulation:** The leaf hashes are sorted by path and concatenated into a single master string, which is then hashed again to produce the `state_hash`.

### 2.2 Why Pruning is Mandatory for Determinism
*   **Timestamp Volatility:** Files like `*.pyc` or `.DS_Store` contain metadata (like timestamps) that changes even if the source code remains identical.
*   **Build Artifact Entropy:** Directory `node_modules/` can contain thousands of files with non-deterministic installation orders. Including them would break the "Deterministic Execution" guarantee.
*   **Security Isolation:** Sensitive files like `.env` must be ignored to prevent their *content hashes* from revealing information about the secrets they contain.

## 3. PROTECTED PATHS: CATEGORICAL BREAKDOWN

### 3.1 Node/Build Artifacts (Structural Noise)
*   `node_modules/`, `dist/`, `build/`: These are large, reproducible artifacts. They are excluded to minimize the "Context Window" load.
*   `__pycache__/`, `*.pyc`, `*.pyo`, `*.pyd`: Python's transient compilation artifacts.

### 3.2 Environment & Persistence (The Master Ledger)
*   `.git/`: Git history is high-entropy and irrelevant to the current *functional state* of the system.
*   `.env`: Security variables. Hashing this would be a security violation.
*   `data/*.db`, `data/*.db-wal`, `data/*.db-shm`: The `cap_events.db` and other SQLite databases are the source of truth. Hashing them *while they are being written to* is a recipe for corruption.

### 3.3 Temporary & Log Files (Runtime Entropy)
*   `tmp/`, `*.log`, `*.bak`, `*.swp`: These files change every second. Including them would trigger constant `ANOMALY_DETECTED` events.
*   `.cap/operator_manifest.json`: This file is frequently updated by the mesh builder. It is excluded from the *filesystem* hash because it is tracked by its own *internal* checksum logic.

## 4. IMPACT ON THE APC SANDBOX & SWARM DISPATCH

### 4.1 Sandbox Construction
When an Intent is dispatched to a remote worker:
1.  **Bundle Generation:** The Executive Node creates a `ContextBundle`.
2.  **Ignore Filtering:** Only files *not* in `.capignore` are included in the bundle.
3.  **Remote Expansion:** The worker receives the bundle and expands it into a clean directory.
4.  **Deterministic Mirroring:** Because of `.capignore`, the worker's sandbox is a perfect semantic mirror of the Executive's state, minus the noise.

### 4.2 Swarm Economy Optimization
Hashing costs compute cycles (CR). Large ignore lists directly reduce the CR cost of an action.
*   **Optimization Strategy:** The PIE Sweeper Agent periodically suggests new ignore patterns (e.g., ignoring large image folders or datasets) to lower the swarm's operational overhead.

## 5. THREAT MODELING & CAUSAL ANOMALIES

### 5.1 The "Blind Spot" Attack
An attacker (or a malfunctioning script) could write malicious code into an ignored directory (e.g., `node_modules/.bin/malicious_shim`).
*   **Detection Failure:** Since the directory is ignored, the APC hasher will not detect the modification.
*   **Defense:** CAP implements a **Shadow Scan** during system idle time, where it performs a full-depth hash (ignoring `.capignore`) and compares it against a baseline "Deep State Hash" stored during v0.2.0 bootstrapping.

### 5.2 Pattern Overlap (The Masking Bug)
If a pattern like `core/` was accidentally added to `.capignore`, the system would become "Blind" to its own kernel.
*   **Self-Healing:** The kernel contains a **Hardcoded Essential List** (e.g., `kernel.py`, `orchestrator.py`) that cannot be ignored. Any attempt to ignore these files will trigger a `SOVEREIGN_POLICY_VIOLATION`.
