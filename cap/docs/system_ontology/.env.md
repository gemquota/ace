# 🔑 SYSTEM ONTOLOGY: `.env`

## 1. EXECUTIVE SUMMARY: THE IDENTITY ANCHOR
The `.env` file is the system's **Environment Configuration and Identity Anchor**. It contains sensitive variables that define the system's unique "Genetic Signature" and its connection to external services. Unlike other configuration files, `.env` is strictly excluded from version control and system state hashing (via `.capignore`) to prevent credential leakage.

In the Cognitive Architecture Platform (CAP), the `.env` file is primarily responsible for the **Genesis Hash anchoring**, which serves as the immutable root of the system's causal history. It ensures that even in a distributed mesh, every node can verify its relationship to the "Master Timeline".

## 2. THE GENESIS HASH (`CAP_GENESIS_HASH`): DEEP DIVE

The `CAP_GENESIS_HASH` is a SHA-256 fingerprint representing the system's state at the moment of its first initialization.

### 2.1 Technical Mechanism
When the `core/clide/kernel/identity.py` script runs for the first time:
1.  **State Snapshot:** It performs a recursive scan of all non-ignored files.
2.  **Fingerprint Generation:** It generates a master hash of the system's source, initial ontology, and configuration.
3.  **Anchoring:** This hash is written to the `.env` file and also recorded in the `system_identity` table of `cap_events.db`.

### 2.2 Role in Causal Consistency
*   **Verification:** Every trace started via `syscalls.cap_trace_start()` includes the current node's `CAP_GENESIS_HASH`.
*   **Tamper Evidence:** If an attacker modifies the source code without going through the proper `AUTopoietic_REFACTOR` protocol, the next state hash will fail to reconcile with the Genesis Hash lineage, triggering an immediate **Sovereign Panic**.
*   **Mesh Integration:** When a new node joins the swarm, it must receive the `CAP_GENESIS_HASH` from the Executive. This hash is then used to verify that the node is joining the "Correct" timeline and not a malicious fork.

### 2.3 The "Identity Crisis" Recovery
If a node's `.env` is corrupted:
*   The kernel will attempt to recover the hash from the `cap_events.db`.
*   If both are lost, the system enters an `UNVERIFIED_STATE`, where all incoming intents are rejected until an operator performs a manual "Identity Reconstruction" via the CLI.

## 3. SECURITY & SECRET VARIABLES: THE VAULT

The `.env` file acts as a localized vault for secrets.

### 3.1 Primary Secrets List
*   **Cognitive APIs:** `OPENAI_API_KEY`, `ANTHROPIC_API_KEY`, `GEMINI_API_KEY`. These are critical for CLIDE's intent compilation and PIE's reasoning.
*   **Mesh Networking:** `TAILSCALE_AUTH_KEY`. Allows the node to join the private Tailscale VPN without human interaction.
*   **Swarm Broker:** `REDIS_PASSWORD`. Secures the multi-node task queue.

### 3.2 Security Constraints & Anti-Leakage
*   **Zero-Prompt Policy:** The `core/clide/memory/` module contains an "Anti-Leakage Filter". This filter scans every outgoing prompt for strings matching values in the `.env` file. If a match is found, the dispatch is aborted with a `SECURITY_LEAK_PREVENTED` error.
*   **APC Sandbox Isolation:** The `.env` file is NOT mounted into the ARM Worker's sandbox. Instead, only the specific subset of environment variables required for the task is injected by the `apc.executor`.

## 4. ENVIRONMENTAL INVARIANTS: NODE BEHAVIOR

### 4.1 Modality Flags
*   `CAP_OPERATIONAL_MODALITY`: Defines the node's behavior mode (e.g., `DISTRIBUTED_HORIZON`, `LOCAL_ONLY`, `SOVEREIGN_AUTONOMY`).
*   `CAP_LOG_LEVEL`: Controls the verbosity of the `logs/` output (e.g., `DEBUG`, `INFO`, `CRITICAL`).

### 4.2 Resource limits (Substrate Specific)
*   `CAP_MAX_CPU`: (Integer) Percentage limit for CPU utilization.
*   `CAP_MAX_RAM`: (Integer) Megabyte limit for memory usage.
*   `CAP_HASHER_THREADS`: (Integer) Number of parallel threads used for the APC state hashing process.

## 5. RECOVERY & INTEGRITY MANAGEMENT

### 5.1 The Ledger-Mirror Fail-safe
The kernel maintains a "Shadow Mirror" of critical `.env` variables in the `cap_events.db`.
1.  **Sync Event:** Every time a secret is updated via a legitimate CLI command, a `SECRET_ROTATED` event is recorded.
2.  **Integrity Check:** During boot, the kernel compares the local `.env` values against the last `SECRET_ROTATED` event hash to ensure the environment hasn't "Drifted" unexpectedly.

### 5.2 Environmental Drifting (Anomaly Detection)
"Environmental Drifting" occurs when a node's local variables diverge from the swarm's expected baseline.
*   **Detection:** PIE monitors the `latency` and `success_rate` of nodes. If a node with specific hardware flags (e.g., `GPU_ENABLED`) suddenly starts failing GPU tasks, PIE will check the node's `.env` for misconfigured paths or missing driver flags.
