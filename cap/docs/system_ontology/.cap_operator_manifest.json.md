# 🧩 SYSTEM ONTOLOGY: `.cap/operator_manifest.json`

## 1. EXECUTIVE SUMMARY: THE SWARM REGISTRY
The `operator_manifest.json` file is the central registry for the **Distributed Swarm Grid**. In the **DISTRIBUTED_HORIZON** modality, CAP no longer assumes a single localized execution environment. Instead, it relies on this manifest to identify active nodes, their hardware capabilities, and their role in the cognitive loop. This file is the "map" that CLIDE uses to decide where an Intent DAG should be dispatched.

The manifest is dynamically updated by the `scripts/build_mesh_state.py` utility, which probes the network (typically via Tailscale) to find and verify active ARM workers. It ensures that the system's "Will" (CLIDE) can find a suitable "Actuator" (APC-RUNTIME) across a diverse hardware landscape.

---

## 2. SCHEMA DEFINITION: FIELD-BY-FIELD ANALYSIS

### 2.1 Metadata & Versioning
*   `manifest_version`: (String) Defines the schema version of the manifest. Essential for forward/backward compatibility as the swarm logic evolves.
*   `system_version`: (String) The current SemVer of the CAP kernel (e.g., `v0.2.1`). Nodes must typically match this version to join the grid.
*   `project_root`: (String) The relative path to the CAP installation (e.g., `dev/cap`).
*   `WORKSPACE_ROOT`: (String) The absolute path on the current node. This is used for hybrid path translation during remote execution.

### 2.2 Entry Points (`entry_points`)
This object maps core system functions to their script paths.
*   `orchestrator`: Path to `core/clide/kernel/orchestrator.py`.
*   `executor`: Path to `core/apc/executor.py`.
*   `inference`: Path to `core/pie/inference.py`.
*   `dashboard`: Path to `core/clide/dashboard/server.py`.
*   `mesh_boot`: Path to `scripts/build_mesh_state.py`.
*   `worker`: Path to the Celery worker entry point (`core.clide.worker`).

### 2.3 Persistence Layer (`persistence`)
Defines the connection strings for the shared state.
*   `events`: (String) SQLite connection string or path for the master ledger (`data/cap_events.db`).
*   `swarm_broker`: (String) The Redis URL used for the task queue. Typically `redis://localhost:6379/0`.
*   `self_model`: (String) Path to `data/cap_arch_model.json`.

## 3. THE ACTIVE NODE REGISTRY (`ACTIVE_NODES`)

The `ACTIVE_NODES` array is a list of objects representing the hardware currently available to the swarm.

### 3.1 Node Identity & Taxonomy
*   `node_id`: (String) Unique identifier (e.g., `executive-termux`). Generated from the node's hostname and a unique hash.
*   `role`: (Enum) Defines the node's primary function in the grid.
    *   `DISPATCHER`: The "Brain". Handles goal injection and compilation.
    *   `HEAVY_COMPUTE`: The "Muscle". Designed for high-CPU/GPU tasks.
    *   `HEALER`: A specialized PIE node focused on anomaly remediation.
*   `ip`: (String) The network address. CAP prefers Tailscale IPs (100.x.y.z) for secure, flat-network communication.

### 3.2 Capability Flags (`capabilities`)
This array defines the specific "skills" or "constraints" of the node.
*   `LIGHTWEIGHT`: Optimized for low-power operation (e.g., Termux on Android).
*   `WINDOWS_NATIVE`: Node is running Windows and can handle `.ps1` or `.bat` tasks.
*   `GPU_ENABLED`: Node has access to CUDA/Vulkan for accelerated inference.
*   `ALWAYS_ON`: Node is a persistent server, ideal for the Redis broker or long-running daemons.

### 3.3 Resource Constraints
*   `load_limit`: (Float, 0.0 to 1.0) The maximum allowed CPU/Memory load. If the node's current load exceeds this, the `DAGScheduler` will bypass it.

## 4. OPERATIONAL LOGIC: SYNCHRONIZATION & HEARTBEATS

### 4.1 The Heartbeat Mechanism
The manifest is a snapshot, but the swarm is dynamic. Heartbeats are managed via Redis:
1.  **Announcement:** A worker starts and sets a key in Redis: `node:heartbeat:<node_id>`.
2.  **Telemetry:** The worker updates this key every 30 seconds with its current `load_avg`.
3.  **Synthesis:** Every 60 seconds, the Executive Node runs `scripts/build_mesh_state.py`, which reads these Redis keys and updates `operator_manifest.json`.

### 4.2 Hybrid Path Abstraction
When a task is dispatched from a Linux Executive to a Windows Execution node, the path `/home/user/dev/cap/file.txt` must be translated.
*   The manifest provides the `WORKSPACE_ROOT` for each node.
*   The `apc.executor` uses this delta to perform dynamic path translation before execution.

## 5. CAUSAL IMPACT & FAILURE MODES

### 5.1 The "Orphaned Task" Scenario
If a node drops off the network (IP timeout), it is removed from the `ACTIVE_NODES` array.
*   **Detection:** The CLIDE Watchdog detects a node marked as `HEAVY_COMPUTE` is missing.
*   **Remediation:** Any tasks `DISPATCHED` to that node that haven't reached `EXEC_COMPLETE` are re-queued with an `INCIDENT_FLAG`.

### 5.2 Manifest Split-Brain
In a network partition, two nodes might both believe they are the "Executive".
*   **Resolution:** CAP uses the **Genesis Hash** and a **Lamport Clock** in the Redis broker. The node with the most recent verifiable ledger entry in `cap_events.db` retains the `DISPATCHER` role.

### 5.3 Error Codes (Manifest Related)
*   `ERR_MANIFEST_PARSE_FAIL`: The JSON is malformed.
*   `ERR_NODE_IP_UNREACHABLE`: A node in the manifest is failing ping/TCP checks.
*   `ERR_CAPABILITY_MISMATCH`: A task was sent to a node that lacks the required flag (e.g., sending a Windows task to a Linux node).

## 6. SECURITY THREAT MODEL

*   **Node Spoofing:** An attacker could inject a fake node into the manifest.
    *   **Defense:** All nodes must provide a `node_secret` (stored in `.env`) that matches the hash recorded in the `cap_events.db` during their first registration.
*   **Manifest Injection:** An attacker modifies the `entry_points` to point to a malicious script.
    *   **Defense:** The `operator_manifest.json` is signed using the Genesis Hash. Any modification without a corresponding `MANIFEST_UPDATE` event will trigger a **Sovereign Panic**.
